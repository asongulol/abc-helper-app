# Track 2 — UI/UX Audit (`abc-helper-app`)

**Scope:** Every page and significant modal/drawer/overlay across the admin surface (`src/app/(admin)/`) and the contractor portal (`src/app/portal/(authed)/`), plus the two login pages and the print views. Read-only audit; no source changed.

**App:** Next.js 16 App Router + React 19 + Tailwind v4 + @supabase/ssr. A payroll admin tool (Hubstaff time → PHP payroll → Wise payouts) plus a contractor self-service portal. The visual system is a hand-ported legacy CSS file ([globals.css](src/app/globals.css), 2,859 lines) keyed to navy + gold, with the portal namespaced under `.portal`.

## §1 Method & standard

### The named standard was absent; impeccable is the substitute

The brief designates `ux-ui-guidelines-v2.md` as the standard. **It does not exist** anywhere on the filesystem (a full-disk `find / -name "ux-ui-guidelines*"` returned nothing). Note that several in-repo components even *cite* a `docs/ux-ui-guidelines.md` — e.g. [EmptyState.tsx:5](src/components/ui/EmptyState.tsx#L5) ("per docs/ux-ui-guidelines.md") and [ConfigClient references] — but **that file is also absent**: `docs/` contains only `CUTOVER-RUNBOOK.md`, `CUTOVER-VERIFICATION.md`, `DEPLOY.md`, `money-core-spec.md`, `RECREATION-HANDOFF.md`, `RECREATION-RECOMMENDATIONS.md`. So the guidelines the code claims to follow are unverifiable.

Per the user's direction, the **impeccable** skill is the governing UX standard for this audit. Reference files read and used:

- `impeccable/SKILL.md` — General rules (Color/contrast ≥4.5:1, Typography, Layout — "cards are the lazy answer", Motion, semantic z-index scale, Absolute bans incl. side-stripe borders).
- `impeccable/reference/product.md` — **primary lens**: this app is admin/dashboard/tool, "design SERVES the product"; earned familiarity; "Modal as first thought… is usually laziness."
- `impeccable/reference/audit.md` — the audit dimensions (A11y/Perf/Responsive/Theming/Anti-patterns) and P0–P3 severity.
- `impeccable/reference/critique.md` — Nielsen heuristics, cognitive-load checklist, persona red flags.
- `impeccable/reference/harden.md` — edge/error/empty/loading states.
- `impeccable/reference/onboard.md` — empty states that teach.
- `impeccable/reference/clarify.md` — UX copy, button labels, confirm-dialog copy.
- `impeccable/reference/layout.md`, `reference/typeset.md`, `reference/colorize.md`, `reference/interaction-design.md` — layout rhythm, type scale, color roles, the eight interactive states + modal `inert`/focus guidance.

WCAG 2.1 AA is applied alongside (contrast, focus order, labels, keyboard nav, landmarks, prefers-reduced-motion).

**Method:** Read the real component + page code and `globals.css`, judged against the references, and verified each cited fact in the file. The static `detect.mjs` detector was not run — it targets HTML/markup or a live URL, and this is a server-rendered JSX/CSS codebase with no built HTML to scan; the primary method (reading code) is what the brief calls for.

## §3 Consolidation candidates

1. **`/calculate` and `/payroll` are the same screen — merge to one route.**
   Both pages render the identical `PayrollShell` component ([calculate/page.tsx](src/app/(admin)/calculate/page.tsx) and [payroll/page.tsx](src/app/(admin)/payroll/page.tsx)). The sidebar lists "Calculate" → `/calculate` ([nav.ts:42](src/components/shell/nav.ts#L42)) while Overview's shortcuts and Process's action links point at `/payroll` ([overview/page.tsx:271](src/app/(admin)/overview/page.tsx#L271)). Net effect: the same workspace lives at two URLs with two names, and the in-app links don't even agree on which one to use. This is a Consistency-and-Standards failure (Nielsen H4). **Merge:** one route (`/payroll`), one nav label. Effort: low.

2. **Fold "Process and Pay" into Payroll/Calculate.**
   [process/page.tsx](src/app/(admin)/process/page.tsx) is a read-only list of locked batches whose only two actions, "Open & pay" and "Unlock", both navigate to `/payroll` ([ProcessShell.tsx:100-141](src/components/process/ProcessShell.tsx#L100)). It adds a hop and a third name for the same period objects. The whole payroll lifecycle (draft → lock → process → pay) could be one route with a state-filter (the pipeline strip on Overview already models exactly these stages). Keep **Batches** separate — it's a genuinely different downstream job (Wise reconciliation). Effort: medium.

3. **Reports "Payout by period" vs. Payroll batch list — dedupe.**
   Reports' first section ([ReportsClient.tsx:150](src/components/reports/ReportsClient.tsx#L150)) renders the same columns (Period, Pay date, Contractors, Net, State) as the Payroll batch list ([PayrollShell.tsx:438-524](src/components/payroll/PayrollShell.tsx#L438)). Make one canonical; have the other link to it. While there, Reports stacks 5 unrelated reports (KPIs, payout-by-period, per-contractor summary, history, weekly activity) in one scroll — split into tabs/sub-routes (layout.md: vary structure, don't monotone-stack). Effort: medium.

4. **Two "add contractor" entry points — pick one.**
   `AddContractorModal` (2-field quick create, [:97](src/components/contractors/AddContractorModal.tsx#L97)) and `AddContractorWizard` (3-step, [:263](src/components/contractors/AddContractorWizard.tsx#L263)) both create a contractor; the wizard's step 1 *is* the quick form. Keep the wizard (it captures hire + onboarding), drop the standalone quick modal, or make the quick modal explicitly "stub now, finish later" with one shared identity step. Effort: low–medium.

5. **Authoring screens scattered across surfaces — centralize under Config.**
   Announcements are authored from a modal on **Contractors** ([ContractorsClient.tsx:324](src/components/contractors/ContractorsClient.tsx#L324)); agreement templates from a modal on **Onboarding** ([OnboardingClient.tsx:261](src/components/onboarding/OnboardingClient.tsx#L261)) *and* presumably from Config. These are configuration concerns. Surface them only from Config's launcher so admins have one place to find "settings-like" authoring. Effort: low.

## §4 Removal candidates

This app is unusually free of dead screens — every route is implemented and maps to a real task. The "removals" are therefore route-collapses (covered in §3) plus genuinely thin/low-value pieces:

1. **`/calculate` route** — remove as a separate route (it's a duplicate alias of `/payroll`; see §3.1). Removal candidate by redundancy, not by being a stub.
2. **`/process` as a standalone route** — low-value pass-through; both actions bounce to `/payroll` (§3.2). Candidate for removal-into-Payroll.
3. **`AddContractorModal`** — redundant with the Wizard (§3.4); remove one of the two.
4. **`DocReminderOverlay` custom sheet** — not removed, but its hand-rolled overlay should be removed in favor of the shared `Modal`/sheet primitive (it reimplements dialog semantics with weaker focus handling; [DocReminderOverlay.tsx:58](src/components/portal/DocReminderOverlay.tsx#L58)).
5. **Phase-2 stub:** Documents file-upload is not yet wired ([DocumentsClient.tsx:167](src/components/documents/DocumentsClient.tsx#L167)) — only metadata is stored. Not a screen to remove; flagged so the "Add" form doesn't imply an upload that doesn't happen.

No "Coming soon" placeholder pages, empty `.stub` routes, or unreachable screens were found. The `.stub` class ([globals.css:483](src/app/globals.css#L483)) is used only for the pre-onboarding gates in the portal (e.g. [sessions/page.tsx](src/app/portal/(authed)/sessions/page.tsx)), which are intentional permission states, not dead ends.

## §5 Accessibility & guideline findings

Grouped by severity. Severity maps: **blocker** = WCAG A failure / blocks a user group; **major** = WCAG AA violation or serious friction; **minor** = polish.

### Blocker

- **B1 — Signature canvas has no accessible name or non-pointer path.** [PortalOnboarding.tsx:512](src/components/portal/PortalOnboarding.tsx#L512). The `<canvas>` has no `aria-label`/`role`, and "Draw signature" is offered before/alongside "Type signature." A keyboard- or screen-reader-only contractor cannot draw. *Mitigation present:* a typed-name fallback exists — make it the **primary** method and label the canvas. **WCAG 2.1.1 (Keyboard), 1.1.1 (Non-text content).** Ref: interaction-design.md §Gesture discoverability ("always provide a visible fallback").

- **B2 — Native `window.confirm()` for a destructive admin action.** [AdminsModal.tsx:113](src/components/shell/AdminsModal.tsx#L113) (remove admin access) and [SessionsClient.tsx:117](src/components/sessions/SessionsClient.tsx#L117) (delete session). `window.confirm` is not stylable, not consistent with the app's ConfirmDangerModal, and gives no consequence context. Inconsistent with every other destructive action in the app, which *does* use the danger modal. **WCAG 3.2.4 (Consistent identification).** Ref: interaction-design.md §Destructive; clarify.md §Confirmation dialogs.

### Major

- **M1 — Tabs are not ARIA tablists, app-wide.** ProfilePanel ([:481-491](src/components/contractors/ProfilePanel.tsx#L481)), PortalProfile ([:332-347](src/components/portal/PortalProfile.tsx#L332)), and onboarding stage tabs render `<button>`s with conditional class (PortalProfile adds `aria-pressed`, ProfilePanel adds nothing). None use `role="tablist"`/`role="tab"`/`aria-selected`/`aria-controls`, so screen-reader users don't get tab semantics or arrow-key roving. **WCAG 4.1.2 (Name, Role, Value).** Ref: interaction-design.md §Roving tabindex.

- **M2 — `div role="button"` for expand/collapse instead of native `<button>`.** PortalTime ([:100](src/components/portal/PortalTime.tsx#L100)), PortalStatements ([:53](src/components/portal/PortalStatements.tsx#L53)), Reports period/contractor rows ([ReportsClient.tsx:258, 687](src/components/reports/ReportsClient.tsx#L258)), Batches `<tr onClick>` ([BatchesClient.tsx:257](src/components/batches/BatchesClient.tsx#L257)). The portal ones add `tabIndex`+`aria-expanded`+key handlers (acceptable but lint-ignored); Reports/Batches expanders have **no** `aria-expanded`/`role`/keyboard handler — mouse-only. **WCAG 2.1.1, 4.1.2.** The Audit Log expander ([AuditTable.tsx:86](src/components/audit/AuditTable.tsx#L86)) does it correctly — replicate that.

- **M3 — Modal focus management does not isolate the background (`inert` missing).** [Modal.tsx](src/components/ui/Modal.tsx) implements a manual JS Tab-cycling trap (lines 56-77) but never sets `inert` on the page behind it, and renders a non-modal `<dialog open>` rather than calling `showModal()`. The interaction-design.md §Modals guidance is explicit: use `inert` on background or native `showModal()`. The manual trap also recomputes focusables on every Tab and can desync when content changes (e.g. the nested ProfilePanel unsaved-guard modal at [ProfilePanel.tsx:1224](src/components/contractors/ProfilePanel.tsx#L1224)). **WCAG 2.4.3 (Focus order).** Ref: interaction-design.md §Modals: The Inert Approach.

- **M4 — `escClose=false` removes the keyboard escape from MiscModal.** [MiscModal.tsx:164](src/components/payroll/MiscModal.tsx#L164). Power/keyboard users cannot dismiss with Escape; they must locate Cancel. interaction-design.md and critique.md (H3 User Control) both require an emergency exit. **WCAG 2.1.2 (No keyboard trap) borderline.**

- **M5 — Color-only status in several status surfaces.** OnboardingDrilldown document-state badges ([:62-67, 858](src/components/onboarding/OnboardingDrilldown.tsx#L62)) convey state via badge *tone* with no text/`aria-label`; Payroll inactive/no-rate rows are background-color-only ([PayrollShell.tsx:794-796](src/components/payroll/PayrollShell.tsx#L794)); Documents row highlight is red/yellow background ([DocumentsClient.tsx:198](src/components/documents/DocumentsClient.tsx#L198) — though its pills carry text, mitigating it); portal Sessions badge tone ([PortalSessions.tsx:17](src/components/portal/PortalSessions.tsx#L17) — text present, OK). **WCAG 1.4.1 (Use of color).** Ref: colorize.md "don't rely on color alone." Where the pill text already names the state, this is fine; the bare row-background cases are not.

- **M6 — Icon-only controls without accessible names.** Payroll revert `↺` ([PayrollShell.tsx:860](src/components/payroll/PayrollShell.tsx#L860)) and Misc-edit have `title` but no `aria-label`; MiscModal delete-row `×` buttons ([:287, 379, 451](src/components/payroll/MiscModal.tsx#L287)) are icon-only; Batches/Reports chevrons lack labels. `title` is not a reliable accessible name. **WCAG 4.1.2.** Ref: interaction-design.md (touch + labels).

- **M7 — Form fields lack `aria-invalid`/`aria-describedby` linkage for errors, app-wide.** The pattern exists in [AddContractorModal.tsx:111](src/components/contractors/AddContractorModal.tsx#L111) (`aria-invalid`) and globals.css styles `[aria-invalid="true"]` ([globals.css:200](src/app/globals.css#L200)), but ProfilePanel, AddContractorWizard, MiscModal, and most portal forms set neither `aria-invalid` nor `aria-describedby` to tie `.field-err` text to its input. Errors are visually adjacent but not programmatically associated. **WCAG 3.3.1/3.3.3.** Ref: harden.md §Form validation; interaction-design.md §Form design.

- **M8 — No `role="progressbar"` on progress affordances.** Onboarding admin progress bar ([OnboardingClient.tsx:91-106](src/components/onboarding/OnboardingClient.tsx#L91)), portal onboarding `.ob-bar` ([globals.css:2120](src/app/globals.css#L2120)), wizard step strip ([AddContractorWizard.tsx:281](src/components/contractors/AddContractorWizard.tsx#L281)). The % text is present (good), but the bar itself is an unlabeled styled div. **WCAG 4.1.2.**

- **M9 — Portal has no skip-to-content link.** Admin shell provides one ([AdminShell.tsx:135](src/components/shell/AdminShell.tsx#L135), `.skip-link` in [globals.css:98](src/app/globals.css#L98)); the portal shell ([PortalShell.tsx:129](src/components/portal/PortalShell.tsx#L129)) does not, so keyboard users tab through the whole nav on every page. **WCAG 2.4.1 (Bypass blocks).**

### Minor

- **m1 — Side-stripe-border family (an Absolute Ban) appears as a deliberate motif.** SKILL.md bans `border-left/right > 1px` as a colored accent. The app uses it intentionally and repeatedly: `.ov-tile { border-left: 4px solid … }` ([globals.css:927](src/app/globals.css#L927)) with tone variants, `.banner { border-left: 5px solid … }` ([globals.css:1203](src/app/globals.css#L1203)), and the DocReminder sheet's `borderTop: 4px gold`. It's a coherent system rather than random AI output, but it's the exact pattern the standard says to rewrite (full borders / background tints / leading icon). Flagged, not blocking.
- **m2 — Toast `z-index: 9999`, outside any semantic scale.** [globals.css:1279](src/app/globals.css#L1279). SKILL.md §Layout: "Never arbitrary values like 999 or 9999; build a semantic z-index scale." The app's other layers are ad-hoc too (modal-bg 20→50 at breakpoints [globals.css:1429,1623,1763](src/app/globals.css#L1429), bottom-nav 40, sticky topbar 10→30). Define `--z-dropdown/sticky/modal/toast`.
- **m3 — `--subtle` (#677083) used for uppercase table headers and group labels.** ([globals.css:25](src/app/globals.css#L25), applied to `th` [:550](src/app/globals.css#L550), `.side-group-label` [:316](src/app/globals.css#L316)). The token comment claims AA 4.5:1; on white it's ~4.6:1 (passes for ≥18px/bold large text, which these are). Verify any place `--subtle` lands on `--surface-2`/`--navy-50` tints where the ratio dips. typeset.md/colorize.md §contrast.
- **m4 — No skeleton states; pages rely on SSR + spinners/`isPending` text.** product.md: "Skeleton states for loading, not spinners in the middle of content." Most pages render fully server-side (acceptable), but mid-action refresh shows a centered Spinner or button text-swaps rather than content-shaped skeletons.
- **m5 — Required fields validated only in JS, not HTML `required`.** e.g. [PortalSessions.tsx:41](src/components/portal/PortalSessions.tsx#L41); form submits then rejects. Add `required`/constraints for error *prevention* (Nielsen H5).
- **m6 — Empty-state inconsistency.** `EmptyState` primitive ([EmptyState.tsx](src/components/ui/EmptyState.tsx)) is used by Contractors/Onboarding/Payroll/SortableTable, but Invoicing ([:317](src/components/invoicing/InvoicingClient.tsx#L317)), Reports (multiple `.empty` divs), Sessions, and Imports use raw `.empty` text — and most empties just say "No X yet" without the teach-the-interface CTA onboard.md asks for.
- **m7 — Modern/classic login toggle gives no current-mode indicator** ([AdminLoginForm.tsx:146](src/components/auth/AdminLoginForm.tsx#L146)). Minor recognition-vs-recall gap.
- **m8 — Print signature `<img alt="signature">` doesn't identify the signer** ([onboarding print page](src/app/(admin)/onboarding/[workerId]/[kind]/print/page.tsx)); say "Contractor signature, Jane Cruz." clarify.md §alt text.

### Positive findings (replicate these)

- **Reduced-motion is genuinely handled** — a global `@media (prefers-reduced-motion: reduce)` kills transitions/animations ([globals.css:2605](src/app/globals.css#L2605)) and several components add targeted overrides (overview stagger delays [globals.css:1192](src/app/globals.css#L1192), nyhero weather FX [:2848](src/app/globals.css#L2848)). ## §6 Cross-cutting UX patterns

**Design-system primitives exist and are largely used.** `src/components/ui/*` provides `Modal`, `ConfirmDangerModal`, `EmptyState`, `Toast`/`useToast`, `Spinner`, `Badge`, `SortableTable`, `PhoneInput`, `EmailInput`, `ContractorPicker`, `useUnsavedGuard` ([index.ts](src/components/ui/index.ts)). Consistency assessment:

- **Buttons:** One vocabulary (`.btn`, `.btn.ghost`, `.btn.danger`, `.btn.sm`) with all eight interactive states defined in CSS (hover/active/focus-visible/disabled, [globals.css:706-768](src/app/globals.css#L706)). Strong consistency. The portal re-skins `.btn` ([globals.css:1961](src/app/globals.css#L1961)) but stays coherent.
- **Toasts:** Single `ToastProvider`/`useToast` ([Toast.tsx](src/components/ui/Toast.tsx)), `role="alert"` for error/warn and `role="status"` otherwise — correct live-region semantics. Used consistently for action feedback.
- **Confirms:** Mostly `ConfirmDangerModal` (good) — **except** the two `window.confirm()` regressions (B2). That's the single biggest consistency break in the destructive-action vocabulary.
- **Tables:** `SortableTable` ([SortableTable.tsx](src/components/ui/SortableTable.tsx)) standardizes sort/filter/empty/card-stack and uses real `<button class="th-sort">` headers with `aria-sort`. But Payroll, Reports, Documents, Sessions, Batches hand-roll their own `<table>`s for inline-edit/expand needs, so the accessible affordances (sortable header buttons, `aria-expanded`) are present in the primitive but absent in the bespoke tables (M2).
- **Forms:** `.field` + label-above convention is consistent; `[aria-invalid]` styling exists in CSS but is applied by almost no component (M7). Labels are mostly real `<label htmlFor>` (good), with a few `.sub`-styled inline `<label>`s in portal forms.
- **Modals:** All centered overlays route through the `Modal` primitive **except** DocReminderOverlay (custom sheet, §4.4) and the two `window.confirm`s. The primitive itself has the `inert` gap (M3). Modal density is high — product.md flags "modal as first thought"; the app reaches for modals for whole workspaces (ProfilePanel 4-tab, OnboardingDrilldown's 9 nested overlays, Config's 7 panels) where routes/sections would carry less cognitive load.
- **Empty/loading/error:** `EmptyState` and `Spinner` exist but adoption is partial (m6); no skeletons (m4); error handling is overwhelmingly toast-based with login pages using inline `role="alert"` banners (good).

Net: the **primitive layer is solid and the visual system is coherent**; the gaps are (a) bespoke tables/overlays bypassing the accessible primitives, and (b) two `window.confirm` defections from the confirm pattern.

## Coverage note

- **Not assessed in a running browser.** All findings are from reading JSX + `globals.css`; I could not measure rendered contrast ratios at actual computed colors, real tab/focus order at runtime, the manual focus trap's live behavior, screen-reader output, or visual layout at breakpoints. The static `detect.mjs` detector was not run (it targets HTML/markup or a live URL; this codebase ships no pre-built HTML to scan and is server-rendered JSX). A live `/impeccable audit` + `/impeccable critique` with browser automation and a contrast pass would confirm the contrast (m3) and focus-trap (M3) findings and could surface responsive-overflow issues I can only infer from CSS.
- **Sub-components partially sampled.** The 12 `config/*` panels and 8 `contractors/*` files were read by exploration agents at summary depth, not every line; I verified the load-bearing claims (ProfilePanel tabs, AdminsModal `window.confirm`, ConfirmDangerModal usages) directly. Deeper per-panel a11y review of each Config card (EmployerCard, ClientsCard, HolidaysCard, etc.) was out of time budget.
- **The two referenced guideline docs (`ux-ui-guidelines-v2.md` and the in-code `docs/ux-ui-guidelines.md`) are both missing**, so claims like "per docs/ux-ui-guidelines.md" in component comments could not be checked against intent — impeccable was used as the substitute standard throughout, per the user's direction.
