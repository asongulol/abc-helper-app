## 4. Cross-cutting conventions

- **Match `money-core-spec.md`'s style**: concrete, source-line-anchored, code-block-heavy.
- Cross-link source paths as repo-relative links; Docusaurus `onBrokenLinks: 'warn'` won't fail
  the build on non-route links.
- Use mermaid for flows (theme is enabled). Keep diagrams small.
- Keep planning/handoff artifacts in `audit/` (out of the published site); only reader-facing
  docs go in `docs/` and `website/reference/`.

## 5. Key decision: OpenAPI vs Server Actions

This app's mutation API is **Next.js Server Actions**, not REST. Only **3 real HTTP endpoints**
exist (2 cron + auth callback). Do **not** fabricate a large OpenAPI spec to look complete —
keep OpenAPI scoped to those 3 (§3.14) and document the real API as a narrative
Server-Actions reference (§3.15). This is deliberate; revisit only if real REST endpoints get added.

## 6. Sidebar implementation

Today `website/sidebars.ts` is `{ type: 'autogenerated', dirName: '.' }` (flat). To get the §2
categories without hand-maintaining order, add `_category_.json` files under `docs/` subfolders
(e.g. `docs/operations/_category_.json` → `{ "label": "Operations", "position": 3 }`) and move
files into those folders — **or** switch `sidebars.ts` to an explicit array. Folder-moving
breaks existing cross-links and the `index.md` link list; if reorganizing, update
`docs/index.md` and add `@docusaurus/plugin-client-redirects` entries
(`website/docusaurus.config.ts:86`) for any moved slugs.

## 7. Acceptance per phase

- `pnpm docs:build` succeeds (it runs `clean-api-docs → gen-api-docs → build`; TypeDoc runs in-build).
- No new broken-link **errors** (warns are fine).
- New pages appear in the sidebar in the intended category/order.
- `docs/index.md` link list updated to include new pages.
- Push to `main` auto-deploys to Vercel (private); spot-check the deployed pages render.
